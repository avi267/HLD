Q. Tell us about a challenging system you built?

**One of the most challenging systems I built was the Supplier Feed batch processing system in my current project**

**Situation:**
We had a requirement to process large volume of batch data related to vehicle key management module. The proposed architecture was tightly coupled, and prone to failures, and inconsistency.

**Task:**
My responsibility was to redesign this system architecture to be scalable, fault-tolerant, and capable of handling batch processing efficiently, while also ensuring data consistency across services.

**Action:**
I designed and implemented an **event-driven microservices architecture** using Azure services:

* Used **Azure Blob Storage** as an intermediate storage layer for large batch files
* Triggered processing using **Azure Functions (Blob triggers)** to handle events asynchronously
* When a supplier uploads a compressed file:

  **A blob-triggered service publishes an event to initiate processing.
  **The feed-preprocessor service consumes the event, performs idempotency checks, decrypts and extracts the file, parses metadata, and persists the processed data into the database.
  **It then emits a batch-level event to the device-batch-manager service, which updates batch-level records and generates device-level events.
  **The device-manager service consumes these events, processes each device, stores device data, and integrates with the external server to generate signed certificates.
  **Once certificates are generated, they are stored in the device_info table, and a response file is created and uploaded back to Azure Blob Storage.

* Implemented the **Outbox pattern** to solve the dual-write problem between database and message broker, ensuring reliable event publishing

For reliability:
* Designed the system to be **idempotent**, so reprocessing wouldn’t cause inconsistencies
* Implemented retry mechanisms and failure handling for downstream services

I also worked on **observability**, integrating logging and distributed tracing to track batch processing and failures effectively.

**Challenges:**

* Handling large batch files without blocking services
* Ensuring consistency between database updates and event publishing
* Managing high throughput without degrading performance
* Debugging intermittent failures in distributed workflows

**Result:**

* Successfully migrated the system to a scalable event-driven architecture
* Eliminated data inconsistency issues using the Outbox pattern
* Increased system reliability and reduced production incidents

**Learning:**
This experience strengthened my understanding of distributed systems, event-driven design, and consistency patterns, which are critical for building scalable financial or real-time processing systems.
